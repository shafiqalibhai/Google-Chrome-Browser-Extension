TYPO3 Browser Compatibility

Introduction

This matrix is not yet complete, but some browsers has been analyzed for compatibility with TYPO3.

MSIE 6.x/WinXP is the reference browser for which everything should be expected to work correctly.

Notes

TYPO3.3b1.9, 5/2002

All functions tested from the Alternative Backend unless they are obvious Classic features.

General notes/rules for HTML:

  • Submitting a form from a <A> tag should be done with document.[formname].submit() and not just submit() (Mac MSIE/5.1)

  • onClick events in image-submit buttons are ignored by some older browsers (and Konqueror).

  • Opera 7 is supported (for Backend click menus) BUT by default it will not work because it identifies itself as "MSIE 6" - and Opera is obviously not MSIE 6 compliant! So set the identification to "Opera" and it will work.

General notes on browser behaviour related to TYPO3


Places where the file-upload field has the width set the ”Browse” button might disappear (collapse)


onClick events in image-type submit buttons are ignored (like Konqueror 2.2.2)

TEXTAREA has no no-wrap property. At least that is NOT working.




Seems slow at login. Suspect the JavaScript MD5() function to be slow.

Seems to reload images on each request (although caching is enabled with default settings). This effect makes it appear slow.

No IMG-alt/title text showing as ”tip” when mouseover!

<textarea> fields cannot contain

onClick events in submit buttons of the image-type ONLY seems to be fired AFTER submission of the form, which means we cannot hold submission back (or change fieldvalues) unless those buttons are ordinary images.


No borderstyles in the tceforms (that is borderstyles of tables do not work).

No IMG-alt/title text showing as ”tip” when mouseover.

Much faster with Alternative Backend.

Slow with tables with many columns. Eg. Web>Access and Web>Info (page tree) is slow.

Serious problems


Apparently if the GET-param ”&params=” is set, this is recognized as a ”&para;” sign! Thus any url with this parameter will pass wrong parameters!

No places in TYPO3 uses &params= anymore, but it should of course be avoided in any future case.



(see TEXTAREA problem)




Even if all data in <textarea>fields are processed through htmlspecialchars() function, a single HTML-comment (with &gt; and &lt; !!) will simply prevent ANYTHING from being displayed in the <textarea> - it will appear like if it is empty. Using htmlentities() is no better. The problem is the char sequence ”&lt;!--”



Blank lines in the beginning of a TEXTAREA field is stripped when content is shown in such a field. Submitted blank lines are stored correctly.
